Final Proposal: Building the Authoritative Knowledge Base (KB-7)


This document outlines the final architectural vision, technology stack, and implementation plan for KB-7. It integrates the robust components from your initial proposals into a cohesive, production-ready system designed for clinical informatics governance, scalability, and semantic precision.


1. Executive Summary


KB-7 will serve as the single source of truth for all clinical terminology, ontologies, and mappings within the ecosystem. It is designed to be a highly available, auditable, and performant service that provides canonical knowledge to downstream systems like the Unified Harmonizer and other knowledge bases.
Its core functions are:
* Centralized Ontology Management: Storing and serving standard ontologies (SNOMED CT, RxNorm, LOINC, ICD-10, AMT).
* High-Fidelity Mapping: Managing complex mappings between different coding systems.
* Robust Governance: Enforcing clinical policy (e.g., "do-not-auto-map" flags) and ensuring full provenance and versioning for every piece of knowledge.
* Advanced Reasoning: Materializing OWL inferences to accelerate queries and uncover implicit relationships in the data.
The entire lifecycle of the knowledge base, from ontology updates to schema changes, will be managed through a GitOps workflow, ensuring reproducibility, auditability, and rigorous clinical sign-off.


2. Architectural Vision: A Hybrid Approach


To balance the semantic power of RDF/OWL with the performance needs of high-throughput mapping, KB-7 will adopt a hybrid data architecture.
1. RDF/OWL Triplestore: This is the heart of the system. It stores the canonical ontologies, their rich relationships (e.g., subClassOf), materialized OWL reasoning results, and all provenance metadata modeled using PROV-O. This is optimized for complex graph traversals and semantic queries (SPARQL).
2. Relational Mapping Database (PostgreSQL): As you correctly identified, a triplestore is not always optimal for simple, high-volume key-value lookups (e.g., "map source code A to target code B"). A dedicated, heavily-indexed PostgreSQL database will store flattened mapping tables for maximum query speed, serving the most common requests from the Unified Harmonizer.
3. API Layer: A microservice will expose both a REST/GraphQL API for simple lookups and a SPARQL endpoint for advanced, federated queries. This layer orchestrates queries between the triplestore and the relational cache.
This hybrid model provides the best of both worlds: deep semantic integrity and reasoning from the triplestore, and operational speed from the relational database.

4. Data Modeling & Implementation Example


All knowledge will be modeled in RDF and serialized as Turtle (.ttl), a human-readable format perfect for versioning in Git.
Here is a concrete example of how a mapping for "Essential hypertension" would be represented, integrating all required components:


Code snippet




# Filename: /mappings/snomed-icd10-hypertension.ttl
# This file is versioned in Git.

@prefix :         <http://your-org.com/kb7/ontology#> .
@prefix owl:      <http://www.w3.org/2002/07/owl#> .
@prefix rdf:      <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix rdfs:     <http://www.w3.org/2000/01/rdf-schema#> .
@prefix xsd:      <http://www.w3.org/2001/XMLSchema#> .
@prefix prov:     <http://www.w3.org/ns/prov#> .
@prefix pav:      <http://purl.org/pav/> .
@prefix skos:     <http://www.w3.org/2004/02/skos/core#> .
@prefix sct:      <http://snomed.info/sct/> .
@prefix icd10cm:  <http://purl.bioontology.org/ontology/ICD10CM/> .

# 1. Policy Flag Declaration (in a core ontology file)
:doNotAutoMap
   a owl:AnnotationProperty ;
   rdfs:label "Do Not Auto-Map" ;
   rdfs:comment "A policy flag indicating that a mapping for this concept requires manual review and should not be automated."@en .

# 2. The Mapping with Provenance and Policy
# This represents the mapping itself as an entity that can be described.
:mapping_icd10_I10_to_snomed_38341003
   a owl:Axiom ;  # Reifying the mapping to attach metadata
   owl:annotatedSource icd10cm:I10 ;
   owl:annotatedProperty skos:exactMatch ;
   owl:annotatedTarget sct:38341003 ;

   # 2a. Provenance (Who, What, When)
   pav:createdBy <http://your-org.com/users/dr-jane-doe> ;
   pav:createdOn "2025-09-17T15:30:00Z"^^xsd:dateTime ;
   pav:version "1.0" ;
   prov:wasDerivedFrom <http://snomed.org/sct/32506021000036107/version/20250731> ;
   rdfs:comment "Mapping established during Q3 2025 terminology review."@en ;

   # 2b. Policy Flag Applied
   # In this case, hypertension mapping is considered safe to automate.
   :doNotAutoMap "false"^^xsd:boolean .

# Example of a sensitive allergy code that should NOT be auto-mapped
:mapping_some_allergy_code
   a owl:Axiom ;
   owl:annotatedSource <http://vendor.com/codes/A123> ;
   owl:annotatedProperty skos:closeMatch ;
   owl:annotatedTarget sct:293425007 ; # Example: Allergy to class of drug
   :doNotAutoMap "true"^^xsd:boolean ; # Policy flag is set to true
   pav:createdBy <http://your-org.com/users/dr-jane-doe> ;
   pav:createdOn "2025-09-17T15:35:00Z"^^xsd:dateTime .



5. GitOps & Clinical Governance Workflow


This workflow ensures every change is automated, validated, and auditable.
1. Propose Change: A clinical informaticist creates a new branch in Git (e.g., feature/add-new-cardiology-mappings).
2. Edit & Commit: They add or modify .ttl files using an editor like Visual Studio Code or Protégé, then commit the changes.
3. Create Pull Request (PR): A PR is created in GitHub/GitLab, targeting the main branch. The PR description details the clinical justification for the change.
4. Automated CI Validation: The PR automatically triggers a CI pipeline:
   * Linting: Checks for valid Turtle syntax.
   * Reasoning Check: robot reason --reasoner HermiT checks for logical inconsistencies (e.g., making a concept a subclass of itself).
   * Constraint Validation: robot validate --shapes policies.shacl runs SHACL rules to enforce governance (e.g., "Every mapping must have a pav:createdBy property").
   * The CI pipeline reports a clear Pass or Fail status on the PR.
5. Clinical Sign-Off: The PR is automatically assigned to designated reviewers (e.g., "Clinical Informatics Lead," "Senior Ontologist"). They review the changes and the justification. A merge is blocked until at least two required approvals are granted.
6. Merge & Deploy: Once approved, the PR is merged into main. This triggers the CD pipeline (ArgoCD/Flux).
7. Automated Deployment: The CD pipeline automatically loads the updated RDF files into a staging GraphDB instance. After automated tests pass, it promotes the changes to the production KB-7, potentially using a blue-green deployment strategy to ensure zero downtime.
