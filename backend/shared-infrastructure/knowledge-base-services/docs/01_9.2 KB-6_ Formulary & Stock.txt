# 1 — Purpose & scope (concise)


**Purpose:** Single source of truth for coverage, cost, prior-auth/step-therapy rules, tiering, and real-time inventory for medication ranking and fulfillment decisions.


**In scope:**


* Plan-specific formulary (coverage, tier, copay, prior auth rules, step-therapy).
* Inventory by location (on-hand, allocated, lot, expiry).
* Fulfilment constraints (restricted distribution, specialty pharmacy).
* Predictive stock forecasts (per location).
* Formulary optimization view for ScoringEngine.


**Out of scope:** clinical safety (KB-4), dosing math (KB-1), guideline evidence (KB-3). (As in your draft.)&#x20;


---


# 2 — High-level architecture & responsibilities


* **ETL (Unified ETL):** ingest PBM feeds, IMS feeds, internal IMS streams; normalize codes with KB-7; validate; harmonize; produce `canonical_formulary` and `canonical_inventory` bundles.
* **KB-6 Service:** exposes gRPC/REST APIs; caches hot data in Redis; indexes search into Elasticsearch; persists canonical data in Postgres.
* **Consumers:** Medication Service (ScoringEngine), Order Fulfillment, Pharmacy UI, Analytics.
* **Governance:** formulary admins maintain overrides, pricing adjustments, and promotion approvals (blue/green).

# 3 — Data model (core tables + DDL snippets)


Primary DB: **Postgres 15+** (partitioning + JSONB). Use `provenance` JSONB on all records.


### 3.1 Key tables (DDL excerpts)


```sql
-- canonical drug identifier is RxNorm (rxnorm:nnnn). Use KB-7 to normalize.
CREATE TABLE formulary (
  id BIGSERIAL PRIMARY KEY,
  dataset_version TEXT NOT NULL,           -- e.g. "pbmA.2025-08.v1" or "kb6.harmonized.2025Q3"
  plan_id TEXT NOT NULL,
  drug_code TEXT NOT NULL,                 -- RxNorm / ATC canonical
  coverage_status TEXT NOT NULL,           -- 'preferred','non_preferred','not_covered'
  tier INTEGER,                            -- numeric tier if present
  copay_cents INTEGER,                     -- integer cents or null
  coinsurance_pct numeric(5,2),
  prior_auth_required boolean DEFAULT false,
  step_therapy jsonb,                      -- structured rules
  restrictions jsonb,                      -- e.g., {specialty:true, distribution:'wholesalerX'}
  effective_from date,
  effective_to date,
  provenance jsonb,                        -- {source_bundle, import_job, kb7_snapshot}
  created_at timestamptz DEFAULT now()
);
CREATE INDEX ix_formulary_plan_drug ON formulary(plan_id, drug_code);


CREATE TABLE inventory (
  id BIGSERIAL PRIMARY KEY,
  dataset_version TEXT NOT NULL,
  location_id TEXT NOT NULL,               -- site / pharmacy / ward
  drug_code TEXT NOT NULL,
  lot_number TEXT,
  qty_on_hand integer NOT NULL DEFAULT 0,
  qty_allocated integer NOT NULL DEFAULT 0,
  qty_reserved integer NOT NULL DEFAULT 0,
  expiry_date date,
  last_updated timestamptz NOT NULL DEFAULT now(),
  provenance jsonb,                         -- {source_system, timestamp, import_job}
  UNIQUE(location_id, drug_code, lot_number)
);
CREATE INDEX ix_inventory_loc_drug ON inventory(location_id, drug_code);


-- materialized view for ScoringEngine (fast single-row read)
CREATE MATERIALIZED VIEW mv_formulary_best AS
SELECT plan_id, drug_code,
       min(copay_cents) as best_copay,
       bool_or(prior_auth_required) as any_prior_auth,
       array_agg(distinct coverage_status) as statuses
FROM formulary
GROUP BY plan_id, drug_code;
CREATE INDEX ix_mv_formulary_best ON mv_formulary_best(plan_id, drug_code);


-- dispensing history for predictive models (partitioned by month)
CREATE TABLE dispensing_history (
  id BIGSERIAL PRIMARY KEY,
  location_id TEXT,
  drug_code TEXT,
  dispensed_qty integer,
  dispensed_at timestamptz,
  patient_age_band TEXT,
  payer_id TEXT,
  deidentified_context jsonb
) PARTITION BY RANGE (dispensed_at);
```


**Notes:** keep `dataset_version` on each row so you can support deterministic rollbacks and audits (blue/green dataset switching).

# 4 — Normalization & ETL contract


* All inbound PBM and inventory codes **must** be normalized to canonical codes using **KB-7** during Normalize stage. ETL must fail the bundle if KB-7 returns unmapped top-frequency items beyond a threshold.
* ETL produces a **Canonical Formulary Bundle (CBF)** and **Canonical Inventory Bundle (CIB)** with a manifest: `{bundle_id, sha256, source_id, kb7_snapshot, record_count, dataset_version}`.
* ETL runs include semantic validations (e.g., copay >= 0, tier in allowed range, prior\_auth rules parse) and produces `validation_report.json`. Failures result in Quarantine.

# 5 — APIs: contracts & example protobufs


Provide both **low-latency** gRPC for hot lookups and **REST/JSON** for admin & bulk queries. Also expose Elasticsearch for search.


### 5.1 gRPC (hot path) — `kb6.proto` (excerpt)


```proto
syntax = "proto3";
package kb6.v1;


message PriceRequest {
  string transaction_id = 1;
  string plan_id = 2;
  string drug_code = 3;       // canonical RxNorm
  string location_id = 4;     // optional: for stock-aware scoring
  repeated string preferences = 5; // e.g., ["preferred_on_formulary"]
}


message PriceResponse {
  string transaction_id = 1;
  string dataset_version = 2;    // dataset used
  bool covered = 3;
  string coverage_status = 4;    // 'preferred','non_preferred','not_covered'
  int32 tier = 5;
  int64 copay_cents = 6;
  double coinsurance_pct = 7;
  bool prior_auth_required = 8;
  repeated string restrictions = 9;
  string provenance_json = 10;   // summarised
}


message StockRequest {
  string transaction_id = 1;
  string location_id = 2;
  string drug_code = 3;
  bool include_lots = 4;
}


message StockResponse {
  string transaction_id = 1;
  string dataset_version = 2;
  int32 qty_on_hand = 3;
  int32 qty_allocated = 4;
  int32 qty_available = 5;   // on_hand - allocated - reserved
  repeated Lot lots = 6;
  string provenance_json = 7;
}


message Lot {
  string lot_number = 1;
  int32 qty_on_hand = 2;
  date expiry_date = 3;
}


service KB6 {
  rpc GetFormularyStatus(PriceRequest) returns (PriceResponse);
  rpc GetStock(StockRequest) returns (StockResponse);
}
```


SLO target: p95 < 40ms for Redis cache hits; p95 < 150ms for SQL fallback.

# 6 — Caching & read patterns


**Redis hot cache** keys (deterministic, version-qualified):


* `kb6:formulary:{dataset_version}:{plan_id}:{drug_code}` → serialized PriceResponse
* `kb6:stock:{dataset_version}:{location_id}:{drug_code}` → StockResponse


**Cache behavior:**


* When dataset cutover occurs, use **namespaced version** swap to immediately invalidate old keys (no mass delete). E.g., `current_dataset_version` pointer in config + cache namespace derived from it.
* Short TTL for dynamic inventory keys (e.g., 5–30s depending on update cadence); longer TTL for formulary (15 minutes) as you proposed. But prefer versioned keys with long TTLs.


**Elasticsearch**: index canonical formulary records for complex search (therapeutic alternatives), mapping includes analyzers and exact filters on `coverage_status`, `tier`, `restrictions`.

# 7 — Formulary evaluation & scoring contract (ScoringEngine)


ScoringEngine calls:


1. `GetFormularyStatus` → returns coverage, copay, prior\_auth flags.
2. `GetStock` (optionally including `include_lots`) → returns availability.
3. ScoringEngine applies weighting (clinical score + cost/availability) and ranks proposals.


**Important:** KB-6 must supply **evidence envelope** (dataset\_version + provenance) so ScoringEngine can log reproducible scoring decisions.

# 8 — Predictive stock service (contract & model)


KB-6 hosts forecast outputs (not necessarily trains models; models can live in Analytics).


**Interface:** daily jobs write `stock_predictions` table:


```sql
CREATE TABLE stock_predictions (
  id bigserial primary key,
  dataset_version text,
  location_id text,
  drug_code text,
  predicted_date date,
  predicted_qty integer,
  model_version text,
  confidence numeric(5,3),
  provenance jsonb
);
CREATE INDEX ON stock_predictions(location_id, drug_code, predicted_date);
```


**API:** `GET /api/v1/stock/predict?location_id=&drug_code=&date=` returns predicted qty and confidence. Used by replenishment and ScoringEngine for “likely availability”.


**Model contract:** model outputs must include `model_version` and `provenance` and be persisted as immutable rows.

# 9 — Versioning, harmonization & promotion (blue/green)


* ETL produces vendor dataset `dataset_version` (e.g., `pbmA.2025-08.v1`). Harmonizer produces `kb6.harmonized.2025Q3.v1`.


* **Promotion:** after QA & governance approval, KB-6 promotes harmonized dataset to `current_dataset_version` via atomic pointer change and publishes `kb6.dataset.promoted` event on Kafka with `dataset_version`, `bundle_sha256`, `effective_at`. Consumers should load new dataset pointer and warm caches.


* **Rollback:** previous dataset is retained; pointer swap restores prior dataset instantly.

# 14 — Failure modes & runbooks


**Mode: ETL failure** → runbook:


1. Check last ETL run logs; restart normalization job; notify data owner.
2. If failure persists > 30 min, alert on-call and promote previous dataset if older dataset missing.


**Mode: Redis cache corruption**:


1. Switch to SQL fallback; rehydrate cache using `current_dataset_version`.
2. Investigate invalidation pattern.


**Mode: stock divergence (cache vs. IMS)**:


1. Mark location as “graceful\_degrade”; block auto fulfillment; alert pharmacy ops.

# 17 — Example workflows (two key flows)


### A. ScoringEngine ranking (hot path)


1. Medication Service creates `proposal` and includes `patient.plan_id` and `preferred_location` (if any).
2. ScoringEngine calls KB-6 `GetFormularyStatus` for each drug + plan (Redis hit expected).
3. ScoringEngine calls KB-6 `GetStock` for the chosen location.
4. ScoringEngine computes score and returns ranked proposals with KB-6 `dataset_version` + `provenance`. Log decision hash.


### B. Order fulfillment decision


1. Order service calls `GetStock` with `include_lots=true`.
2. If qty\_available >= required, allocation is done (qty\_allocated increment via transaction), and inventory record updated through ETL/stream.
3. If insufficient, fallback: check `stock_predictions` for expected arrival; or call alternative location endpoints.
