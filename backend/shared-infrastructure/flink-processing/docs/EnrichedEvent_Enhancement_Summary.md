## New Fields Added to EnrichedEvent

### Module 2 Enrichment Fields

#### 1. **immediateAlerts** (List<SimpleAlert>)
- **Purpose**: Threshold-based alerts generated by Module 2
- **Examples**:
  - Heart rate > 140 bpm
  - Systolic BP < 90 mmHg
  - Temperature > 38.3°C
  - SpO2 < 92%
- **Populated by**: Module 2 (Context Assembly & Enrichment)
- **Default**: Empty list (no immediate concerns)

#### 2. **riskIndicators** (RiskIndicators)
- **Purpose**: Structured boolean flags for CEP pattern matching
- **Key Features**:
  - Replaces manual vital extraction in CEP patterns
  - Enables efficient pattern matching: `event.riskIndicators.tachycardia && event.riskIndicators.hypotension`
  - Contains 30+ clinical indicators organized by category:
    - Vital sign concerns (tachycardia, hypotension, fever, hypoxia, etc.)
    - Lab abnormalities (elevated lactate, creatinine, glucose issues, etc.)
    - Medication concerns (vasopressors, anticoagulation, insulin, etc.)
    - Clinical context (ICU, post-operative, immunosuppressed, etc.)
    - Trend indicators (heart rate, blood pressure, creatinine trends)
    - Timing indicators (time since admission, last vitals, last labs)
- **Populated by**: Module 2 `buildRiskIndicators()` method
- **Default**: null (populated on enrichment)

#### 3. **clinicalScores** (Map<String, Double>)
- **Purpose**: Calculated clinical scoring systems
- **Key Examples**:
  - `"mews_score"` → 5.0 (Modified Early Warning Score)
  - `"qsofa_score"` → 2.0 (Quick SOFA)
  - `"news2_score"` → 7.0 (National Early Warning Score 2)
  - `"sofa_score"` → 3.0 (Sequential Organ Failure Assessment)
- **Populated by**: Module 2 `calculateClinicalScores()` method
- **Default**: Empty map

### Module 3 Semantic Enrichment Fields

#### 4. **potentialInteractions** (List<DrugInteraction>)
- **Purpose**: Drug interactions detected by semantic enrichment
- **Key Features**:
  - Uses existing `DrugInteraction` class
  - Contains: drug1, drug2, severity, mechanism, risk score
  - Supports intervention recommendations
- **Populated by**: Module 3 (Semantic Enrichment) after knowledge base consultation
- **Default**: Empty list (no interactions detected)

#### 5. **applicableProtocols** (List<String>)
- **Purpose**: Clinical protocols relevant to patient state
- **Examples**:
  - "sepsis_bundle"
  - "aki_prevention"
  - "vte_prophylaxis"
  - "glycemic_control"
- **Populated by**: Module 3 (Semantic Enrichment)
- **Default**: Empty list

## Data Flow Architecture

```
Module 1: Event Ingestion
    ↓
Module 2: Context Assembly & Enrichment
    ↓ (Generates)
    - immediateAlerts (threshold checks)
    - riskIndicators (CEP pattern flags)
    - clinicalScores (MEWS, qSOFA, NEWS2, SOFA)
    ↓
Module 3: Semantic Enrichment
    ↓ (Adds)
    - potentialInteractions (drug safety)
    - applicableProtocols (evidence-based care)
    ↓
Module 4: CEP Pattern Matching
    ↓ (Consumes riskIndicators for complex patterns)
    - Uses structured flags instead of manual parsing
    - Efficient pattern matching: event.riskIndicators.tachycardia
```

## Builder Pattern Enhancements

### New Builder Methods

**Batch setters:**
- `immediateAlerts(List<SimpleAlert>)` - Set all alerts
- `riskIndicators(RiskIndicators)` - Set risk indicators
- `clinicalScores(Map<String, Double>)` - Set all scores
- `potentialInteractions(List<DrugInteraction>)` - Set all interactions
- `applicableProtocols(List<String>)` - Set all protocols

**Incremental setters:**
- `addImmediateAlert(SimpleAlert)` - Add single alert
- `addClinicalScore(String, Double)` - Add single score
- `addPotentialInteraction(DrugInteraction)` - Add single interaction
- `addApplicableProtocol(String)` - Add single protocol

### Usage Example

```java
EnrichedEvent event = EnrichedEvent.builder()
    .id("evt-123")
    .patientId("patient-456")
    .eventType(EventType.VITAL_SIGNS)
    .eventTime(System.currentTimeMillis())

    // Module 2 enrichment
    .addImmediateAlert(
        SimpleAlert.builder()
            .alertType("HIGH_HEART_RATE")
            .severity(AlertSeverity.WARNING)
            .message("Heart rate 125 bpm exceeds threshold")
            .build()
    )
    .riskIndicators(
        RiskIndicators.builder()
            .tachycardia(true)
            .fever(true)
            .heartRateTrend(TrendDirection.INCREASING)
            .build()
    )
    .addClinicalScore("mews_score", 5.0)
    .addClinicalScore("qsofa_score", 1.0)

    // Module 3 enrichment
    .addPotentialInteraction(interaction)
    .addApplicableProtocol("sepsis_bundle")

    .build();
```

## Design Decisions

### 1. **Separate Alert Types**
- **Decision**: Split into `immediateAlerts` (simple thresholds) vs CEP patterns (complex logic)
- **Rationale**: Clear separation of concerns, simpler threshold logic in Module 2, complex patterns in Module 4

### 2. **Structured Risk Indicators**
- **Decision**: Use boolean flags instead of parsing payload in CEP patterns
- **Rationale**:
  - Performance: Pre-computed flags faster than runtime parsing
  - Clarity: `event.riskIndicators.tachycardia` more readable than `event.payload.get("heartRate") > 100`
  - Maintainability: Centralized logic in Module 2

### 3. **Map for Clinical Scores**
- **Decision**: Use `Map<String, Double>` instead of dedicated fields
- **Rationale**:
  - Flexibility: Easy to add new scoring systems
  - Extensibility: Module 2 can calculate any score without model changes
  - Standardization: Consistent access pattern

### 4. **Empty Collections as Defaults**
- **Decision**: Initialize lists/maps to empty instead of null
- **Rationale**:
  - Null safety: Prevents NullPointerException in consumers
  - Backward compatibility: Existing code gets empty collections, not null
  - Cleaner code: No null checks required for iteration

### 5. **AlertSeverity Migration Strategy**
- **Decision**: Keep deprecated values, add new ones
- **Rationale**:
  - Backward compatibility: Existing code continues to work
  - Clear migration path: Deprecated annotations guide updates
  - Gradual transition: Teams can migrate at their own pace
