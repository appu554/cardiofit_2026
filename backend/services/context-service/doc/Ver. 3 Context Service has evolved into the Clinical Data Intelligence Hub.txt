Context Service: Comprehensive Design & Implementation Guidelines
Executive Overview
The Context Service has evolved into the Clinical Data Intelligence Hub of our platform, serving as the single source of truth for assembled clinical context. It bridges the gap between raw distributed data and actionable clinical intelligence, supporting both the Calculate → Validate → Commit pattern and the Digital Reflex Arc with sub-100ms response times.
Core Architecture Evolution
The Context Service as Clinical Data Federator
The Context Service operates as a Clinical Data Orchestra:
* Conductor of distributed data sources
* Composer of clinical context recipes
* Curator of data quality and freshness
* Guardian of clinical data governance
* Accelerator through intelligent caching

Module 3: Intelligent Caching Architecture
3.1 Multi-Layer Cache Design
Three-Tier Caching Strategy
L1: Workflow Context Cache (In-Process)
├── Scope: Single workflow execution
├── Duration: Workflow lifetime (5-10 min)
├── Storage: Local memory
├── Size: ~100MB per workflow
├── Hit Ratio Target: >95%
└── Use Cases:
    ├── Repeated context access
    ├── Step-to-step sharing
    ├── Proposal variations
    └── Validation iterations


L2: Distributed Cache (Redis Cluster)
├── Scope: Cross-workflow, per-patient
├── Duration: 15-30 minutes
├── Storage: Redis with persistence
├── Size: ~10GB total
├── Hit Ratio Target: >80%
└── Use Cases:
    ├── Common patient data
    ├── Recent calculations
    ├── Frequent recipes
    └── Session continuity


L3: Service-Level Caches
├── Scope: Domain-specific data
├── Duration: Service-defined
├── Storage: Service memory/Redis
├── Size: Varies by service
├── Hit Ratio Target: >60%
└── Use Cases:
    ├── Reference data
    ├── Terminology mappings
    ├── Historical aggregates
    └── Computed values
