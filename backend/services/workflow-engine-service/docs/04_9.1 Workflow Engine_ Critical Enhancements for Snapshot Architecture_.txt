# Workflow Engine: Critical Enhancements for Snapshot Architecture


After reviewing the comprehensive Workflow Engine design against our new Medication Service architecture with Recipe Resolution and Snapshot Management, here are the necessary enhancements:


## 1. Snapshot-Aware Orchestration


### Current Gap
The Workflow Engine design doesn't explicitly handle the new snapshot-based data consistency model.


### Required Enhancement


```typescript
// workflow-engine/orchestration/snapshot-orchestrator.ts


export class SnapshotAwareOrchestrator {
  async executeCalculatePhase(
    command: ClinicalCommand,
    workflowInstance: WorkflowInstance
  ): Promise<ProposalWithSnapshot> {
    // Key change: Workflow Engine no longer manages context
    // It simply passes intent to Medication Service
    const proposal = await this.medicationService.generateProposal(
      command.intent,
      command.patientId
    );
    
    // Store snapshot reference in workflow state for audit
    workflowInstance.state.snapshotReference = proposal.snapshotReference;
    workflowInstance.state.snapshotId = proposal.snapshotReference.snapshotId;
    
    return proposal;
  }
  
  async executeValidatePhase(
    proposal: ProposalWithSnapshot,
    workflowInstance: WorkflowInstance
  ): Promise<ValidationResult> {
    // Safety Gateway now uses snapshot from proposal
    const verdict = await this.safetyGateway.validate(
      proposal,
      workflowInstance.patientId
    );
    
    // Verify same snapshot was used
    if (verdict.evidenceEnvelope.snapshotId !== proposal.snapshotReference.snapshotId) {
      throw new SnapshotConsistencyError('Different snapshots used for calculation and validation');
    }
    
    return verdict;
  }
}
```

## 3. Override Learning Loop Integration


### Complete the Learning Loop


```typescript
// workflow-engine/learning/override-analyzer.ts


export class OverrideLearningLoop {
  private kafkaProducer: KafkaProducer;
  private analyticsService: AnalyticsService;
  
  async captureOverride(
    workflowId: string,
    override: ClinicalOverride,
    snapshot: SnapshotReference
  ): Promise<void> {
    const overrideEvent: OverrideEvent = {
      eventId: generateId(),
      workflowId,
      timestamp: new Date(),
      
      // Clinical context
      override: {
        type: override.type,
        reason: override.reason,
        clinician: override.clinicianId,
        originalVerdict: override.originalVerdict,
        overriddenTo: override.newDecision
      },
      
      // Snapshot for reproducibility
      snapshotId: snapshot.snapshotId,
      snapshotChecksum: snapshot.checksum,
      
      // Additional context
      patientFactors: await this.extractPatientFactors(snapshot),
      rulesFired: override.rulesOverridden,
      
      // Learning metadata
      learningPriority: this.calculateLearningPriority(override)
    };
    
    // Send to learning pipeline
    await this.kafkaProducer.send({
      topic: 'clinical-overrides',
      messages: [{
        key: workflowId,
        value: JSON.stringify(overrideEvent)
      }]
    });
    
    // Real-time analytics
    await this.analyticsService.trackOverride(overrideEvent);
  }
  
  async analyzePatterns(): Promise<OverridePatterns> {
    // Aggregate override patterns
    const patterns = await this.analyticsService.query({
      metric: 'override_patterns',
      groupBy: ['rule', 'medication', 'clinician_role'],
      timeRange: 'last_30_days'
    });
    
    // Identify learning opportunities
    const opportunities = patterns.filter(p => 
      p.overrideRate > 0.3 || // High override rate
      p.count > 100 // High volume
    );
    
    return {
      patterns,
      learningOpportunities: opportunities,
      recommendedActions: this.generateRecommendations(opportunities)
    };
  }
}
```

## 5. Performance Optimizations


### Snapshot-Aware Caching


```typescript
// workflow-engine/cache/workflow-cache.ts


export class WorkflowCache {
  private snapshotCache: Map<string, SnapshotReference> = new Map();
  
  async cacheSnapshotReference(
    workflowId: string,
    snapshot: SnapshotReference
  ): Promise<void> {
    // Cache snapshot reference for workflow duration
    this.snapshotCache.set(workflowId, snapshot);
    
    // Set TTL based on workflow SLA
    setTimeout(() => {
      this.snapshotCache.delete(workflowId);
    }, 300000); // 5 minutes
  }
  
  async getSnapshotReference(workflowId: string): Promise<SnapshotReference | null> {
    return this.snapshotCache.get(workflowId) || null;
  }
}
```

## 7. Error Handling Updates


### Snapshot-Specific Error Handling


```typescript
class SnapshotErrorHandler {
  async handleSnapshotErrors(error: Error, workflowInstance: WorkflowInstance): Promise<void> {
    if (error instanceof SnapshotExpiredError) {
      // Snapshot expired between calculate and validate
      // Need to restart workflow with fresh snapshot
      await this.restartWorkflowWithFreshSnapshot(workflowInstance);
      
    } else if (error instanceof SnapshotIntegrityError) {
      // Critical: snapshot tampering detected
      await this.escalateToSecurity(error, workflowInstance);
      await this.failWorkflowSafely(workflowInstance);
      
    } else if (error instanceof SnapshotNotFoundError) {
      // Snapshot missing - check if it was archived
      const archived = await this.checkArchive(error.snapshotId);
      if (archived) {
        await this.retrieveFromArchive(error.snapshotId);
      } else {
        await this.recreateSnapshot(workflowInstance);
      }
    }
  }
}
```

## Summary of Critical Changes


1. **Workflow Engine becomes simpler** - No longer manages context fetching
2. **Snapshot tracking throughout** - Every phase tracks snapshot usage
3. **Learning loop formalized** - Override patterns feed back to rules
4. **Error handling enhanced** - Snapshot-specific error scenarios
5. **Monitoring expanded** - Snapshot and learning metrics added


These enhancements ensure the Workflow Engine properly orchestrates the new snapshot-based architecture while maintaining its role as a pure orchestrator that doesn't handle business logic or data management directly.
