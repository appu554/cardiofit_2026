# Final Proposal: Medication Service Architecture with Recipe & Snapshot Management


## Executive Summary


The Medication Service requires significant architectural enhancements to support the new Recipe Resolution and Snapshot Management paradigm. The service now owns recipe resolution internally and orchestrates snapshot creation through the Context Gateway, ensuring complete data consistency across the Calculate → Validate → Commit pattern.


## 1. Revised Core Architecture


### 1.1 Key Architectural Changes


```typescript
// medication-service/architecture.ts


interface MedicationServiceArchitecture {
  // NEW: Recipe Resolution is now internal
  recipeResolver: RecipeResolver;
  
  // NEW: Direct integration with Context Gateway for snapshots

  // EXISTING: Knowledge Base access via Apollo Federation
  apolloFederation: ApolloFederationClient;
  
  // EXISTING: Core calculation engines
  candidateBuilder: CandidateBuilder;
  doseCalculationEngine: RustDoseEngine;
  scoringEngine: ScoringEngine;
  
  // EXISTING: Persistence layer
  proposalRepository: ProposalRepository;
  outboxPublisher: OutboxEventPublisher;

```


### 1.2 Updated Service Boundaries


The Medication Service now has expanded responsibilities:
- **Owns Recipe Resolution** (previously external)
- **Orchestrates Snapshot Creation** (via Context Gateway)
- **Includes Snapshot Reference** in all proposals
- **Maintains Recipe Definitions** as part of its domain


## 2. Enhanced 4-Phase Workflow Implementation


### Phase 1: Ingestion & Recipe Resolution (ENHANCED)


```typescript
class MedicationService {
  private recipeResolver: RecipeResolver;
  
  async phase1_ingestAndResolve(
    request: MedicationRequest
  ): Promise<{manifest: IntentManifest, recipe: WorkflowRecipe}> {
    // Step 1: Apply Orchestrator Rule Base (ORB)
    const intentManifest = await this.orchestratorRuleBase.classify(
      request.indication,
      request.clinicalContext
    );
    
    // Step 2: Resolve complete recipe (NEW - internal)
    const recipe = this.recipeResolver.resolveWorkflowRecipe(
      intentManifest.protocolId,
      intentManifest.contextNeeds
    );
    
    // Recipe includes ALL fields needed for:
    // - Medication calculation (demographics, labs)
    // - Safety validation (allergies, conditions)
    // - Audit requirements (provider, encounter)
    
    return { manifest: intentManifest, recipe };
  }
}
```


### Phase 2: Context Assembly via Snapshot (TRANSFORMED)

class MedicationService {

## 3. Recipe Resolver Implementation


### 3.1 Internal Recipe Management


```typescript
class RecipeResolver {
  private recipeDefinitions: Map<string, RecipeTemplate>;
  
  resolveWorkflowRecipe(
    protocolId: string,
    contextNeeds: ContextNeeds
  ): WorkflowRecipe {
    const template = this.recipeDefinitions.get(protocolId);
    
    // Merge requirements from different phases
    const calculationFields = this.getCalculationFields(protocolId);
    const safetyFields = this.getSafetyFields(protocolId);
    const auditFields = this.getAuditFields();
    
    // Add conditional fields based on context
    const conditionalFields = this.resolveConditionalFields(
      contextNeeds,
      template.conditionalRules
    );
    
    return {
      recipeId: `${protocolId}_${timestamp}`,
      requiredFields: this.mergeAndDeduplicate([
        calculationFields,
        safetyFields,
        auditFields,
        conditionalFields
      ]),
      freshnessRequirements: template.freshnessRules,
      ttlSeconds: this.calculateTTL(protocolId),
      allowLiveFetch: false, // Default to false for safety
      allowedLiveFields: [] // Exceptional cases only
    };
  }
}
```

## 5. Error Handling & Resilience


### 5.1 Snapshot-Specific Error Handling


```typescript
class MedicationServiceErrorHandler {
  async handleSnapshotErrors(error: Error): Promise<void> {
    if (error instanceof SnapshotCreationError) {
      // Recipe might be invalid or data sources unavailable
      await this.fallbackToMinimalRecipe();
    } else if (error instanceof SnapshotExpiredError) {
      // Cannot proceed with expired snapshot
      throw new ClinicalSafetyError('Snapshot expired, restart workflow');
    }
  }
}
```

## 7. Critical Implementation Requirements


### 7.1 Mandatory Changes


1. **Recipe Resolver Integration**
   - Must be implemented as internal module
   - Recipe definitions stored and versioned locally
   - Support for conditional field resolution


2. **Snapshot Orchestration**
   - Service MUST call Context Gateway for snapshot creation
   - Never fetch data directly from sources
   - Always include snapshot reference in proposals


3. **Knowledge Base Queries**
   - Use Apollo Federation for all KB access
   - Never embed clinical rules in service code
   - Cache KB responses appropriately

## Summary


This architecture ensures the Medication Service:


1. **Owns Recipe Resolution** - No external dependencies for recipe logic
2. **Orchestrates Snapshots** - Ensures data consistency via Context Gateway
3. **Leverages Knowledge Bases** - All clinical logic externalized
4. **Maintains Audit Trail** - Every decision traceable to specific snapshot
5. **Enables Safe Validation** - Safety Gateway uses same snapshot data


The key insight is that the Medication Service has evolved from a simple calculator to an intelligent orchestrator that manages the complete lifecycle of medication proposals while ensuring data consistency through immutable snapshots.
