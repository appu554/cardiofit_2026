1. Service Overview & Design Principles
The Intake & Onboarding Service (M0) is the clinical front door of the Vaidshala metabolic correction engine. It handles three critical functions: patient enrollment (identity creation, ABHA linking, consent collection, tenant assignment, co-enrollee registration), structured clinical history collection (50 slots across 8 clinical domains via conversational WhatsApp flow or app-based structured forms), and real-time safety screening (11 HARD_STOP rules and 8 SOFT_FLAG rules evaluated deterministically as each slot is filled). The service also manages biweekly check-in sessions (M0-CI) that reuse the same slot infrastructure with a reduced 12-slot subset.

1.1 Why Go for M0
M0’s core workload is state machine traversal, deterministic safety rule evaluation, and event-sourced slot storage. These are CPU-bound, branching-heavy operations where Go’s compiled performance, predictable latency (no GC pauses in hot paths), and goroutine-per-session model excel. Each active WhatsApp conversation is a goroutine with ~2KB stack. At peak enrollment (corporate wellness program launch: 500 patients enrolling in one day), M0 handles 500 concurrent stateful sessions without thread pool exhaustion. The safety engine must evaluate rules in under 5ms per slot fill — Go’s compiled code delivers this consistently. Python would require careful optimization to hit these latencies under concurrent load.

1.2 Immutable Design Principles
1. ZERO LLM CLINICAL DECISIONS. The LLM handles Hindi language understanding only (extracting medication names from free text, parsing numeric values from conversational sentences). Every safety rule, slot validation, state transition, and clinical threshold is deterministic compiled Go code. The LLM is never in the decision path for patient safety.
2. EVENT-SOURCED SLOT STORAGE. Every slot fill creates a new immutable event row (patient_id, slot_name, value, confidence, source, timestamp). No updates. No deletes. A current_slots view returns the latest event per slot. This gives the audit trail the Clinical Protocol demands and the temporal reconstruction the Knowledge OS requires.
3. SAFETY ENGINE RUNS SYNCHRONOUSLY ON EVERY SLOT FILL. There is no batch safety check. When slot S04 (fbg_recent) is filled with value 380, the safety engine evaluates HARD_STOP H2 (FBG > 300) within the same goroutine, within the same request. The patient receives the safety message before the next question is asked. Latency budget: <5ms for safety evaluation.
4. PROTOCOL AUTHORITY RULE. Where M0 Technical Spec and Clinical Protocol conflict on any clinical threshold, safety trigger, or drug interaction rule, the Clinical Protocol is authoritative. M0 is engineering implementation. The Protocol is clinical authority.

2. Three Enrollment Channels
The intake service handles three distinct enrollment journeys, each with different acquisition triggers, identity verification, consent flows, and onboarding UX. All three converge on the same 50-slot clinical history collection and safety screening engine.


3. Enrollment State Machine
The enrollment process is a finite state machine with 8 states. Each state has entry conditions, actions, and transition rules. The state machine is the same for all three channels — the channel affects which states are mandatory and the medium of interaction, not the state logic itself.


4. Clinical Slot Table: 50 Slots Across 8 Domains
Every data point collected during intake maps to a named slot. Slots are typed, classified (mandatory/conditional/optional), safety-evaluated, and protocol-referenced. The slot table is the contract between M0 and all downstream KB services. KB-20 stores slot events. KB-22 uses slot values for monitoring node initialization. KB-23 uses slots for initial decision card generation. KB-26 uses slots for twin state initialization.

SLOT COMPLETION RULE: Intake session CANNOT transition to INTAKE_COMPLETED until: (1) All mandatory (M=Y) slots filled with confidence HIGH, (2) All safety (S=Y) slots pass Safety Engine evaluation, (3) Any HARD_STOP triggered → session enters HARD_STOPPED, physician alerted. Slots marked ‘Derived’ in source column are extracted programmatically from other slots, not asked directly.

5. Safety Engine (Deterministic, No LLM)
The Safety Engine evaluates every slot fill synchronously. It is compiled Go code with zero external dependencies. No LLM, no ML model, no network call. Rule evaluation must complete within 5ms. Rules are organized into two tiers: HARD_STOPs (session paused, physician alerted) and SOFT_FLAGs (documented, pharmacist aware, session continues).

5.1 HARD_STOP Rules (11 Rules)
HARD_STOP = Session pauses immediately. Patient receives standardized safety message in their language. Physician alerted via FCM push + SMS within 30 minutes. Session cannot resume without physician clearance.


5.2 SOFT_FLAG Rules (8 Rules)
SOFT_FLAG = Documented in pharmacist review. Session continues. Pharmacist addresses during review. No patient message (patient is not alarmed).

The graph is loaded from a YAML definition file — adding new slots means adding nodes and edges to the definition, not writing new traversal logic. The State Manager is a generic graph traversal engine.

6.3 NLU Confidence Handling
Confidence ≥ 0.90: Accept value. Fill slot. Proceed to next node.
Confidence 0.70–0.89: Accept value but ask confirmation: ‘Aapne [value] bataya. Sahi hai?’ [Haan | Nahi]. If Nahi: re-ask slot.
Confidence < 0.70: Do not fill slot. Re-ask with simplified phrasing. If second attempt also < 0.70: escalate to pharmacist for manual extraction from raw message.
The LLM is NEVER trusted blindly. Every LLM extraction is confidence-scored. Below 0.70, the system does not guess — it asks again or escalates to a human. This is the fundamental safety boundary between AI language understanding and clinical data entry.

7. Biweekly Check-in State Machine (M0-CI)
The 14-day lifestyle clock (Protocol §10) triggers a biweekly check-in that collects 12 slots in 7 states. M0-CI reuses M0’s slot infrastructure, NLU pipeline, Safety Engine, and WhatsApp delivery. It does NOT duplicate any component. M0-CI produces a Trajectory Signal (not a Decision Card) that feeds M4’s correction loop classification.


7.1 M0-CI Output: Trajectory Signal
The check-in produces a Trajectory Signal consumed by M4 (Correction Loop Orchestrator). The signal classifies the patient into one of four trajectories:
STABLE_IMPROVEMENT: FBG trending down, adherence >60% across behaviors, no safety flags. Action: continue current protocol.
FRAGILE_IMPROVEMENT: FBG improving but adherence inconsistent (<60% in one or more behaviors). Action: BCE adjusts coaching technique, no protocol change.
PROTOCOL_FAILURE: FBG not improving or worsening despite adequate adherence (>60%). Action: lifestyle ceiling may be reached. KB-23 generates Decision Card for physician (medication escalation review).
DISENGAGEMENT: Missed check-in, adherence <30%, or multiple missed actions. Action: pharmacist outreach call. If continued: escalation to physician for re-engagement decision.

8.3 Concurrency Model
Each active enrollment session (patient actively answering questions) runs as a goroutine. Session state is persisted in PostgreSQL (crash-safe) with a Redis distributed lock (prevents concurrent message processing for the same patient). WhatsApp webhooks are retry-prone — the idempotency layer (Redis hash: messageId → processed, 24hr TTL) ensures the same message is never processed twice. At peak corporate enrollment (500 patients in one day), the service handles ~500 concurrent goroutines with ~1MB total goroutine stack overhead. Each slot fill involves: Redis dedup check (1ms) → Redis session lock acquire (1ms) → PostgreSQL slot event insert (5ms) → safety engine evaluation (1–5ms) → next question determination (1ms) → WhatsApp message send (50–200ms, external). Total per-slot latency: ~60–210ms, dominated by the external WhatsApp API call.

8.4 Offline Enrollment (Government Channel)
ASHA field enrollment must work without internet connectivity. The ASHA tablet adapter (internal/asha/) implements a local SQLite queue that stores enrollment sessions offline. When connectivity returns, the sync module replays queued sessions to the server. Conflict resolution: server-side slot events are authoritative. If the same patient was partially enrolled by ASHA offline and a pharmacist entered data online (unlikely but possible), the server merges by taking the most recent slot event per slot. ABHA verification (requires ABDM API) is deferred until connectivity is available — the patient can proceed through intake offline, but ENROLLED state requires ABHA verification for government channel.

8.5 Deployment
Container: Single Go binary in Alpine Docker image (~20MB). Stateless except for Redis session locks.
Kubernetes: Deployment with HPA (min 2, max 10 replicas). Memory-based scaling (each concurrent session ~4KB).
Dependencies: PostgreSQL (slot events, sessions), Redis (locks, dedup, NLU cache), Kafka (event publishing), WhatsApp Cloud API, ABDM Sandbox/Prod API, Anthropic API (NLU, via ingestion service proxy for rate limiting).

FINAL ARCHITECTURAL NOTE: M0 is the clinical front door. Every patient who enters the correction loop passes through M0’s safety engine. If M0 misses a HARD_STOP, the patient enters the protocol with an undetected safety risk. If M0’s slot extraction is inaccurate, KB-26 initializes the twin with wrong data. If M0’s session management drops a patient, they disappear from the funnel. Build M0 with the same rigor as the safety engine it contains: deterministic, auditable, zero-tolerance for silent failures.

Mode | Description | Latency | Accuracy | Used For
BUTTON | WhatsApp interactive buttons (2–3 options). No NLU needed. Direct mapping. | <50ms | 100% (deterministic) | Categorical slots: dinner_pattern, protein_intake, walk_days, yes/no questions. Preferred mode wherever possible.
REGEX | Pattern matching on numeric values. Regex extracts digits from message text. | <10ms | 95%+ (fails on non-numeric responses) | Numeric slots: fbg_recent, weight, waist, BP values. Handles: ‘132’, ‘sugar 132 tha’, ‘132 mg’.
NLU (LLM) | LLM extracts structured data from free-text Hindi. Used ONLY when buttons and regex are insufficient. | 200–500ms | 85–92% (confidence scored) | Medication list extraction (complex, free-text names + dosages), HbA1c with date (‘7.4 tha 2 mahine pehle’), complex symptom descriptions.
DEVICE | Value auto-populated from connected device (BP monitor, glucometer, wearable) via app relay. | <100ms | 95%+ (device accuracy) | BP, glucose, HR, weight — when patient has a connected device. Bypasses conversational question entirely.
ASHA | ASHA worker enters value on tablet. Structured form input. No NLU needed. | <50ms | 98%+ (trained worker) | Government channel: physical measurements (weight, waist, BP taken by ASHA). ASHA form validates input ranges.
